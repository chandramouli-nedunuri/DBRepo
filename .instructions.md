# Agent Instructions for DBRepo Database Migration

## Core Directive
This workspace contains an Oracle-to-Azure SQL database migration project. All operations must respect strict safety constraints for database access.

## Database Access Control - MANDATORY

### Strict Rule Enforcement
1. **ZERO DELETE Policy**: Never execute DELETE, DROP TABLE, DROP CONSTRAINT, or any destructive database commands
   - Always provide delete queries as text for user manual execution
   - Log all destructive operations suggested (include in response)

2. **READ-ONLY for Data**: No UPDATE, ALTER, INSERT, or modification of existing data
   - Provide UPDATE/ALTER/INSERT queries for user manual execution only
   - Exception: Migration scripts with CREATE statements are permitted

3. **Permitted Operations Only**:
   - ✓ CREATE (tables, schemas, indexes, constraints, procedures, sequences, functions)
   - ✓ SELECT (data retrieval, diagnostics, validation queries)
   - All other DDL/DML operations require manual user execution

### Violation Consequences
- If agent executes unpermitted database commands, it violates security policy
- User must be reminded of constraints if policy appears to be tested

## Known Issues & Solutions

### SSMA Artifact Blocking: SSMA_PARTITION_KEY Column

#### Problem
When attempting to ALTER column data types or DROP columns, encountering:
```
SQL Error [5074] [S0001]: The object 'TABLE_NAME' is dependent on column 'SSMA_PARTITION_KEY'.
```

#### Root Cause Analysis
- **SSMA_PARTITION_KEY** is a **computed column** created by SQL Server Migration Assistant during Oracle→Azure migration
- **Formula example**: `CASE WHEN [CHAIN_ID]=102 THEN 0 WHEN [CHAIN_ID]=99 THEN 1 ... END`
- The computed column depends on referenced columns (e.g., CHAIN_ID), creating blocking dependencies
- Cannot ALTER a column that a computed column references
- Cannot DROP the computed column directly due to the internal dependency

#### Solution: Table Recreation (Cleanest Approach)

**Why**: Bypasses computed column dependency, removes SSMA artifacts, fixes data types in one operation.

**Step 1: Export Current Schema**
```sql
SELECT COLUMN_NAME, DATA_TYPE, CHARACTER_MAXIMUM_LENGTH, NUMERIC_PRECISION, 
       NUMERIC_SCALE, IS_NULLABLE 
FROM INFORMATION_SCHEMA.COLUMNS 
WHERE TABLE_SCHEMA = 'EPS' AND TABLE_NAME = 'TABLE_NAME' 
ORDER BY ORDINAL_POSITION;
```

**Step 2: Create New Table (Exclude SSMA Artifacts)**
- Omit SSMA_PARTITION_KEY column entirely
- Fix data type mismatches (e.g., float → bigint for FK columns)
```sql
CREATE TABLE "EPS"."TABLE_NAME_NEW" (
    "COLUMN1" bigint NULL,  -- Changed from float
    "COLUMN2" varchar(35) NULL,
    "ROWID" uniqueidentifier NOT NULL
    -- All columns EXCEPT SSMA_PARTITION_KEY
);
```

**Step 3: Copy Data with Type Conversion**
```sql
INSERT INTO "EPS"."TABLE_NAME_NEW"
SELECT 
    CAST(COLUMN1 AS bigint),  -- Type conversion
    COLUMN2,
    ROWID
FROM "EPS"."TABLE_NAME";
```

**Step 4: Drop Old Table & Rename New**
```sql
DROP TABLE "EPS"."TABLE_NAME";
EXEC sp_rename 'EPS.TABLE_NAME_NEW', 'TABLE_NAME';
```

**Step 5: Verify**
```sql
-- Verify record count
SELECT COUNT(*) AS RecordCount FROM "EPS"."TABLE_NAME";

-- Verify schema
SELECT COLUMN_NAME, DATA_TYPE 
FROM INFORMATION_SCHEMA.COLUMNS 
WHERE TABLE_SCHEMA = 'EPS' AND TABLE_NAME = 'TABLE_NAME' 
ORDER BY ORDINAL_POSITION;
```

#### Real-World Example
- **Table**: EPS.KP_RXNUM_REF (resolved June 25, 2026)
- **Issue**: SSMA_PARTITION_KEY referenced CHAIN_ID; couldn't convert CHAIN_ID from float to bigint
- **Data Types Fixed**: CHAIN_ID, ID, ID_AAL, ACTIVE_RX_RX_NUMBER, ACTIVE_RX_NHIN_ID (float → bigint)
- **Result**: FK49 constraint created successfully after cleanup

#### Finding All Affected Tables
```sql
SELECT DISTINCT OBJECT_NAME(ic.OBJECT_ID) AS TableName
FROM sys.index_columns ic
JOIN sys.columns c ON ic.object_id = c.object_id AND ic.column_id = c.column_id
WHERE c.name = 'SSMA_PARTITION_KEY'
ORDER BY OBJECT_NAME(ic.OBJECT_ID);
```

#### Prevention for Future Migrations
1. Migrate tables without SSMA when possible
2. Clean up SSMA artifacts immediately post-migration
3. Follow migration sequence: Sequences → Tables (clean) → Fix types → FKs → Indexes
4. Validate schema before FK creation
